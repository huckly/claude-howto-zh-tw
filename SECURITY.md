# 安全政策

## 概述

Claude How To 專案的安全性對我們來說非常重要。本文件概述了我們的安全實踐，並描述如何負責任地回報安全漏洞。

## 支援版本

我們為以下版本提供安全更新：

| 版本 | 狀態 | 支援期限 |
|------|------|---------|
| 最新版（main） | 積極支援 | 當前版本 + 6 個月 |
| 1.x 版本 | 積極支援 | 直到下一個主要版本 |

**注意**：作為教育指南專案，我們專注於維護最新的最佳實踐和文件安全性，而非傳統的版本支援。更新直接應用於 main 分支。

## 安全實踐

### 程式碼安全

1. **依賴管理**
   - 所有 Python 依賴都固定在 `requirements.txt` 中
   - 透過 dependabot 和手動審查定期更新
   - 每次提交都使用 Bandit 進行安全掃描
   - 使用 pre-commit hooks 進行安全檢查

2. **程式碼品質**
   - 使用 Ruff 進行 Lint 以捕獲常見問題
   - 使用 mypy 進行型別檢查以防止型別相關漏洞
   - Pre-commit hooks 強制執行標準
   - 所有變更在合併前都需經過審查

3. **存取控制**
   - `main` 分支啟用分支保護
   - 合併前需要審查
   - 合併前狀態檢查必須通過
   - 限制倉庫的寫入權限

### 文件安全

1. **範例中不包含機密**
   - 範例中的所有 API 金鑰都是佔位符
   - 認證資訊永不寫死
   - `.env.example` 檔案展示所需的變數
   - 關於機密管理有清楚的警告

2. **安全最佳實踐**
   - 範例展示安全的模式
   - 文件中明確標示安全警告
   - 提供官方安全指南的連結
   - 在相關章節討論認證處理

3. **內容審查**
   - 所有文件都經過安全問題審查
   - 貢獻指引中包含安全注意事項
   - 驗證外部連結和參考

### 依賴安全

1. **掃描**
   - Bandit 掃描所有 Python 程式碼的漏洞
   - 透過 GitHub 安全警報進行依賴漏洞檢查
   - 定期手動安全稽核

2. **更新**
   - 安全修補程式迅速套用
   - 主要版本謹慎評估
   - 更新日誌包含安全相關更新

3. **透明度**
   - 安全更新記錄在提交中
   - 漏洞揭露以負責任的方式處理
   - 適當時發布公開安全通告

## 回報漏洞

### 我們關注的安全問題

我們感謝以下方面的回報：
- 腳本或範例中的**程式碼漏洞**
- Python 套件中的**依賴漏洞**
- 任何程式碼範例中的**密碼學問題**
- 文件中的**認證/授權缺陷**
- 配置範例中的**資料暴露風險**
- **注入漏洞**（SQL、指令等）
- **SSRF/XXE/路徑遍歷**問題

### 超出範圍的安全問題

以下不在本專案的範圍內：
- Claude Code 本身的漏洞（請向 Anthropic 回報）
- 外部服務或函式庫的問題（請向上游回報）
- 社會工程或使用者教育（不適用於本指南）
- 沒有概念驗證的理論性漏洞
- 透過官方管道回報的依賴漏洞

## 如何回報

### 私有回報（推薦）

**對於敏感的安全問題，請使用 GitHub 的私有漏洞回報：**

1. 前往：https://github.com/luongnv89/claude-howto/security/advisories
2. 點擊「Report a vulnerability」
3. 填寫漏洞詳情
4. 包含：
   - 漏洞的清楚描述
   - 受影響的元件（檔案、章節、範例）
   - 潛在影響
   - 重現步驟（如適用）
   - 建議的修正（如果你有的話）

**接下來會發生什麼：**
- 我們會在 48 小時內確認收到
- 我們會調查並評估嚴重性
- 我們會與你合作開發修正方案
- 我們會協調揭露時間表
- 我們會在安全通告中致謝（除非你希望匿名）

### 公開回報

對於非敏感問題或已公開的問題：

1. **開啟 GitHub Issue** 並標記 `security` 標籤
2. 包含：
   - 標題：`[SECURITY]` 後接簡短描述
   - 詳細描述
   - 受影響的檔案或章節
   - 潛在影響
   - 建議的修正

## 漏洞回應流程

### 評估（24 小時）

1. 我們確認收到回報
2. 我們使用 [CVSS v3.1](https://www.first.org/cvss/v3.1/specification-document) 評估嚴重性
3. 我們判斷是否在範圍內
4. 我們與你聯繫，提供初步評估

### 開發（1-7 天）

1. 我們開發修正方案
2. 我們審查和測試修正
3. 我們建立安全通告
4. 我們準備發行說明

### 揭露（依嚴重性而定）

**嚴重（CVSS 9.0-10.0）**
- 立即發布修正
- 發布公開通告
- 提前 24 小時通知回報者

**高（CVSS 7.0-8.9）**
- 48-72 小時內發布修正
- 提前 5 天通知回報者
- 發布時公開通告

**中（CVSS 4.0-6.9）**
- 在下次定期更新中發布修正
- 發布時公開通告

**低（CVSS 0.1-3.9）**
- 包含在下次定期更新中
- 發布時通告

### 發布

我們發布的安全通告包含：
- 漏洞描述
- 受影響元件
- 嚴重性評估（CVSS 分數）
- 修正版本
- 替代方案（如適用）
- 回報者致謝（經許可）

## 回報者的最佳實踐

### 回報前

- **驗證問題**：你能一致地重現嗎？
- **搜尋現有 issues**：是否已被回報？
- **查閱文件**：是否有安全使用的指引？
- **測試修正**：你建議的修正是否有效？

### 回報時

- **具體說明**：提供確切的檔案路徑和行號
- **包含上下文**：為什麼這是安全問題？
- **展示影響**：攻擊者可以做什麼？
- **提供步驟**：我們如何重現？
- **建議修正**：你會如何修正？

### 回報後

- **耐心等待**：我們的資源有限
- **保持回應**：快速回答後續問題
- **保持機密**：在修正前不要公開揭露
- **尊重協調**：遵循我們的揭露時間表

## 安全標頭和配置

### 倉庫安全

- **分支保護**：main 分支需要 2 個批准才能變更
- **狀態檢查**：所有 CI/CD 檢查必須通過
- **CODEOWNERS**：關鍵檔案指定審查者
- **簽章提交**：建議貢獻者使用

### 開發安全

```bash
# 安裝 pre-commit hooks
pre-commit install

# 在本地執行安全掃描
bandit -c pyproject.toml -r scripts/
mypy scripts/ --ignore-missing-imports
ruff check scripts/
```

### 依賴安全

```bash
# 檢查已知漏洞
pip install safety
safety check

# 或使用 pip-audit
pip install pip-audit
pip-audit
```

## 貢獻者的安全指引

### 撰寫範例時

1. **永不寫死機密**
   ```python
   # 不好的做法
   api_key = "sk-1234567890"

   # 好的做法
   api_key = os.getenv("API_KEY")
   ```

2. **警告安全影響**
   ```markdown
   **安全注意**：永遠不要將 `.env` 檔案提交到 git。
   立即加入 `.gitignore`。
   ```

3. **使用安全的預設值**
   - 預設啟用認證
   - 在適用處使用 HTTPS
   - 驗證和清理輸入
   - 使用參數化查詢

4. **記錄安全注意事項**
   - 解釋為什麼安全很重要
   - 展示安全 vs 不安全的模式
   - 連結到權威來源
   - 明顯地包含警告

### 審查貢獻時

1. **檢查暴露的機密**
   - 掃描常見模式（api_key=、password=）
   - 審查配置檔案
   - 檢查環境變數

2. **驗證安全程式碼實踐**
   - 沒有寫死的認證資訊
   - 適當的輸入驗證
   - 安全的認證/授權
   - 安全的檔案處理

3. **測試安全影響**
   - 這可以被濫用嗎？
   - 最壞情況是什麼？
   - 是否有邊界情況？
