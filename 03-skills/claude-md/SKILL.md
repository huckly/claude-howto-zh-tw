## 使用者輸入

```text
$ARGUMENTS
```

在繼續之前，您**必須**先考量使用者輸入（若非空白）。使用者可指定：
- `create` — 從頭建立新的 CLAUDE.md
- `update` — 改善現有的 CLAUDE.md
- `audit` — 分析並報告目前 CLAUDE.md 的品質
- 特定路徑，用於建立或更新（例如 `src/api/CLAUDE.md`，用於目錄專屬指示）

## 核心原則

**LLM 是無狀態的**：CLAUDE.md 是唯一會自動納入每次對話的檔案，是 AI agents 進入程式碼庫的主要上手文件。

### 黃金守則

1. **少即是多**：最前沿的 LLM 能遵循約 150-200 條指示。Claude Code 的系統提示已使用約 50 條。請保持 CLAUDE.md 的聚焦與簡潔。

2. **普遍適用性**：只納入與每次對話都相關的資訊。任務專屬指示應放在獨立檔案中。

3. **別讓 Claude 充當 Linter**：樣式指引會佔用情境且降低指示遵循品質，請改用確定性工具（prettier、eslint 等）。

4. **永遠不要自動生成**：CLAUDE.md 是 AI 工作流的最高槓桿點，應謹慎手動撰寫。

## 執行流程

### 1. 專案分析

首先分析目前的專案狀態：

1. 檢查現有的 CLAUDE.md 檔案：
   - 根目錄：`./CLAUDE.md` 或 `.claude/CLAUDE.md`
   - 目錄專屬：`**/CLAUDE.md`
   - 全域使用者設定：`~/.claude/CLAUDE.md`

2. 識別專案結構：
   - 技術堆疊（語言、框架）
   - 專案類型（monorepo、單一應用、函式庫）
   - 開發工具（套件管理器、建置系統、測試執行器）

3. 檢視現有文件：
   - README.md
   - CONTRIBUTING.md
   - package.json、pyproject.toml、Cargo.toml 等

### 2. 內容策略（WHAT、WHY、HOW）

圍繞三個維度組織 CLAUDE.md：

#### WHAT — 技術與結構
- 技術堆疊概覽
- 專案組織（對 monorepo 特別重要）
- 關鍵目錄及其用途

#### WHY — 目的與背景
- 專案的功能為何
- 為何做出某些架構決策
- 各主要元件的職責

#### HOW — 工作流程與慣例
- 開發工作流程（bun vs node、pip vs uv 等）
- 測試程序與指令
- 驗證與建置方法
- 關鍵的「注意事項」或非直觀需求

### 3. 漸進式揭露策略

對於較大型的專案，建議建立 `agent_docs/` 資料夾：

```
agent_docs/
  |- building_the_project.md
  |- running_tests.md
  |- code_conventions.md
  |- architecture_decisions.md
```

在 CLAUDE.md 中，以如下方式參照這些檔案：
```markdown
如需詳細建置說明，請參閱 `agent_docs/building_the_project.md`
```

**重要**：使用 `file:line` 參照而非程式碼片段，以避免情境過時。

### 4. 品質限制

在建立或更新 CLAUDE.md 時：

1. **目標長度**：300 行以下（最好在 100 行以內）
2. **無樣式規則**：移除任何 linting／格式化指示
3. **無任務專屬指示**：移至獨立檔案
4. **無程式碼片段**：改用檔案參照
5. **無冗餘資訊**：不要重複 package.json 或 README 中已有的內容

### 5. 必要章節

結構良好的 CLAUDE.md 應包含：

```markdown
# 專案名稱

簡短一行描述。

## 技術堆疊
- 主要語言及版本
- 關鍵框架／函式庫
- 資料庫／儲存（若有）

## 專案結構
[僅適用於 monorepo 或複雜結構]
- `apps/` — 應用程式進入點
- `packages/` — 共用函式庫

## 開發指令
- 安裝：`指令`
- 測試：`指令`
- 建置：`指令`

## 關鍵慣例
[僅限非直觀、高影響力的慣例]
- 慣例 1 及簡短說明
- 慣例 2 及簡短說明

## 已知問題 / 注意事項
[持續讓開發者踩坑的問題]
- 問題 1
- 問題 2
```

### 6. 應避免的反模式

**請勿包含：**
- 程式碼樣式指引（請使用 linter）
- 如何使用 Claude 的說明文件
- 顯而易見模式的冗長解釋
- 複製貼上的程式碼範例
- 通用最佳實踐（「撰寫乾淨的程式碼」）
- 特定任務的指示
- 自動生成的內容
- 大量的 TODO 清單

### 7. 驗證清單

在完成前，請確認：

- [ ] 300 行以下（最好 100 行以內）
- [ ] 每一行都適用於所有對話
- [ ] 無樣式／格式化規則
- [ ] 無程式碼片段（使用檔案參照）
- [ ] 指令已驗證可運作
- [ ] 複雜專案已使用漸進式揭露
- [ ] 關鍵注意事項已記錄
- [ ] 與 README.md 無冗餘

## 輸出格式

### `create` 或預設：

1. 分析專案
2. 依上述結構起草 CLAUDE.md
3. 呈現草稿供審閱
4. 獲批准後寫入適當位置

### `update`：

1. 讀取現有 CLAUDE.md
2. 對照最佳實踐進行稽核
3. 識別：
   - 應移除的內容（樣式規則、程式碼片段、任務專屬內容）
   - 應精簡的內容
   - 缺漏的必要資訊
4. 呈現變更供審閱
5. 獲批准後套用變更

### `audit`：

1. 讀取現有 CLAUDE.md
2. 生成報告，包含：
   - 目前行數 vs 目標行數
   - 普遍適用內容的百分比
   - 發現的反模式清單
   - 改善建議
3. **不要**修改檔案，僅提供報告

## AGENTS.md 處理

若使用者要求建立或更新 AGENTS.md：

AGENTS.md 用於定義專業化的 agent 行為。與 CLAUDE.md（用於專案情境）不同，AGENTS.md 定義：
- 自訂 agent 角色與能力
- Agent 專屬指示與限制
- 多 agent 情境的工作流程定義

套用相似原則：
- 保持聚焦與簡潔
- 使用漸進式揭露
- 參照外部文件而非嵌入內容

## 注意事項

- 納入前務必確認指令可正常運作
- 若有疑慮，不如省略 — 少即是多
- 系統提示告訴 Claude，CLAUDE.md「可能相關也可能不相關」— 雜訊越多，越容易被忽略
- Monorepo 最能受益於清晰的 WHAT/WHY/HOW 結構
- 目錄專屬的 CLAUDE.md 應更加聚焦
